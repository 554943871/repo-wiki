# Page Page Guidance

Page 页面解释用户可见页面、入口、导航和页面状态。它关注用户能看到什么、从哪里进入、如何跳转，以及页面如何承接 flow、module 和 model。

## 应该帮助读者回答

- 这个页面服务哪个用户目标或业务场景。
- 用户从哪里进入这个页面。
- 页面可以跳转到哪里，什么时候返回。
- 页面有哪些关键可见状态、空态、异常态或禁用态。
- 页面依赖哪些 flow、module 和 model。
- 哪些路由、接口、组件或代码位置能帮助验证页面行为。

## 适合写

- 页面目标和入口。
- 页面导航和返回路径。
- 用户可见状态和关键展示条件。
- 相关 flows、modules 和 models。
- 理解页面行为所需的关键路由、接口或数据来源。
- 仓库相对代码锚点。

## 避免写

- 完整 DOM 树。
- CSS、样式 token 或组件库细节。
- 组件清单式盘点。
- 后端调用链。
- 每个字段的展示规则全集。
- 把页面当成 module 的能力边界替代品。

## 推荐表达

Page 页面可以自然组合：

- 页面定位。
- Entry points。
- Navigation。
- Visible states。
- Related flows。
- Related modules。
- Related models。
- Code anchors。

这些是写作建议，不是固定字段。

## LLM 语义检查问题

- 页面是否从用户可见行为出发，而不是从组件树或路由表出发？
- 入口和跳转是否能被代码、路由、产品材料或用户确认支撑？
- 页面状态是否区分了业务状态、接口状态和 UI 展示态？
- 是否链接到相关 flow、module 和 model，而不是重复它们的完整内容？
- 是否避免把后端调用链或模块职责写进页面页？
