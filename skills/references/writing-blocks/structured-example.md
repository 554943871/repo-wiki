# Structured Example Block

Structured Example Block 用于 Model Family Page 的成员 model mock data。它帮助读者在定义句之后立即看见 model 的字段组成、关键字段含义、关系锚点和边界提示。

它不是字段全集、数据库结构、DTO 清单、真实生产记录或格式完整性证明。

## 使用时机

在 `wiki/05-models/<topic>.md` 的 `成员模型` 章节中使用。每个成员 model 默认应在醒目定义句之后紧跟一个精简 mock data 代码块。

适合使用：

- 读者需要快速看懂一个 model 的字段大概长什么样。
- 字段含义会影响 model 定义、关系、生命周期、事实来源或协作规则。
- 关系图里出现的相邻 model 需要一个关系锚点级字段片段帮助读者理解它为什么在图里。

不适合使用：

- 只是为了让页面看起来完整。
- 没有足够证据支撑字段名或字段含义。
- 需要展示的内容已经退化成字段全集、存储结构或原始数据 dump。

## 推荐形态

成员 model 的局部顺序固定为：

1. `### <ModelName>`
2. `**定义：** ...`
3. `jsonc` mock data
4. `补充说明`

示例：

````md
### <ModelName>

**定义：** 一句话说明这个 model 表达什么事实、在当前 model family 里负责什么。

```jsonc
{
  "stable_field": "mock value",   // 稳定字段的示例值
  "related_field": {              // 与当前 model 关系相关的字段组
    "nested_field": "mock value"  // 嵌套字段的示例值
  }
}
```

补充说明：

- 只解释示例之外仍必要的边界、事实来源、关系或不确定性。
````

## 代码块规则

- 必须使用 `jsonc`，不要使用未注释的 `json` 代码块。
- 每个可见字段 key 必须有行尾 `//` 注释，包括容器字段和嵌套字段。
- 不要求给 `{`、`}`、数组元素本身或没有字段 key 的行加注释。
- 同一个代码块内，`//` 注释应形成稳定的视觉列。
- 如果字段名或 mock value 太长导致注释破列，优先缩短 mock value、拆分结构、减少字段，或把字段移到补充说明。
- 数组对象里的可见字段 key 也必须注释；如果多个对象会导致重复注释，只保留一个代表性对象或裁剪示例。

## 证据与裁剪

示例字段只能来自已有 wiki facts、直接 repo evidence、已确认用户说明，或当前页面正文已经支撑的事实。

如果证据不足，不要为了组成一个完整示例而补造字段。可以在补充说明中写“示例只展示已确认字段”，或把缺口报告为 uncertainty / coverage gap。

字段选择应服务于读者理解：

- 保留能解释 model 定义、关系、生命周期、事实来源或协作规则的字段。
- 删除只增加体积、不改变理解的字段。
- 用短 mock value 表达含义，避免复制真实数据或过长值。

## 常见反模式

- 用未注释的 `json` dump 替代解释。
- 为了对齐或完整感补造字段。
- 把 mock data 写得像完整结构或真实记录。
- 让行尾注释参差不齐，导致读者无法快速扫字段含义。
- 在示例前后重复一张完整字段表，造成同一信息出现两套表达。
