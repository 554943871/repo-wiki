# Viewport-Readable Whitebox Layout Policy

Whitebox rendering uses a Viewport-Readable Layout Policy instead of treating square diagrams as the goal: the renderer should pick the generated SVG that is easiest to read at a fixed browser or Markdown viewport width while preserving semantic correctness. This is implemented as renderer-owned Multiple Candidate Layout Selection: the renderer may try rightward layered layouts first, wrapped rightward layouts for overly wide diagrams, and downward layered layouts only as a fallback when rightward candidates are materially less readable.

Consequences: `.whitebox.yaml` remains a topology-only Diagram Source Model and does not gain aspect-ratio, direction, wrapping, coordinate, or candidate-selection fields. Candidate layouts are rejected when they fail layout quality checks such as missing nodes, overlap, incorrect connector endpoints, unreadable labels, or ports not straddling owner boundaries; remaining candidates are scored by viewport readability rather than by strict mathematical squareness. Generated SVG may include Layout Candidate Metadata so later agents can understand the selected renderer policy, chosen candidate, readability score, and display shape without treating those details as source-model facts. Fixture checks should lock down the behavior boundary, such as rejecting source-level layout fields, exposing selected-candidate metadata, and preferring readable output for overly wide diagrams, without treating exact coordinates or score values as stable contracts.
