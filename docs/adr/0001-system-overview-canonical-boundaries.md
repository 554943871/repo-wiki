# System Overview Canonical Boundaries

System overview guidance uses hard canonical-boundary rules instead of soft reader-value heuristics: a C2 Main Runtime Unit must have an independent runtime process boundary, a C1 External System must be an actual external service/platform/system rather than an SDK/adapter/protocol, canonical names must be single stable names with one owner index, and Runtime Topology entity nodes must reuse C1 or C2 canonical names 100%. This keeps generated repo-local wikis from promoting interface layers, in-process capabilities, local files, implementation adapters, or temporary topology labels into stable system concepts.

Consequences: C1/C2 classification rules live as writing guidance in `skills/references/writing-guidance/system-overview.md`, not as target-wiki output sections; protocols, read/write modes, files, persistence media, modules, pages, flows, models, decisions, and infrastructure details belong in edge labels, short notes, or continue-reading routes unless they first become confirmed C1/C2 canonical concepts. Public surfaces, pages, models, flows, decisions, SDKs, adapters, local files, and in-process objects are not promoted into C1/C2; old wiki content should be reclassified without losing information. `wiki/01-system.md` uses `运行拓扑（Runtime Topology）` as the reader-facing section title; `Deployment` is not used as the default heading. Runtime Topology must still be drawn as a Mermaid flowchart, but it only draws confirmed nodes and edges; gaps and unclear relationships are stated after the diagram instead of being guessed.
