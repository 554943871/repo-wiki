# Whitebox Component Diagram Source Model

Canonical module owner pages use Whitebox Component Diagrams as their required module-boundary diagram, with a topology-only Diagram Source Model as the modifiable fact source and generated SVG as the reader-facing rendering. Confirmed C2 runtime units are the first-level root module owner pages when drawn as an enclosing Whitebox component. Stable subsystems, code modules, or other lower-level modules become enclosing Whitebox components only when tied to a C2 root module or upper owner page. The source model records only confirmed components, parts, ports, interface roles, connectors, externals, and evidence; it does not record coordinates, layout hints, unresolved questions, confidence states, prose explanations, contract summaries, or generated image details.

This deliberately avoids treating Mermaid, PlantUML, draw.io XML, or Markdown prose as the diagram truth. Those formats can be useful renderings or sketches, but the wiki skill suite needs diagrams that agents can regenerate and understand from explicit semantics, while renderer-owned layout algorithms handle grouping, ordering, routing, canvas size, UML port/interface notation, and complexity warnings.

Consequences: every canonical module owner page must include a Whitebox Component Diagram, and even an empty whitebox must show at least one confirmed boundary port connected to an external node. Former module-map or overview pages that use Whitebox must either be confirmed as canonical module owner pages or remain blocked as `meaning_loss_risk`; the suite no longer treats ownerless overview boundaries as a separate page family. Connectors are directed and typed as `external`, `delegation`, `assembly`, or `interfaceAssembly`; connectors must terminate at legal endpoints rather than whole components, and `interfaceAssembly` connects required/provided interface roles instead of ports themselves. Dense or tangled renderings are preserved as complete views and may be noted as a Diagram Complexity Signal, but they are not automatic refactor verdicts. Dense diagrams may also have additional Derived Whitebox Views generated from the same Diagram Source Model, but those views never replace the complete view and never become separate fact sources. Derived views are selected automatically by renderer rules, not by `views` or layout-selection fields in the source model. Generated derived SVGs may exist under the page-local `assets/` directory as artifacts, while module owner pages embed only the derived views that answer a clear reader question or materially improve understanding. Markdown beside the diagram may include `内部模块` tables and one contract table per boundary port, but those tables supplement reader understanding and never become diagram fact-source fields. The `.whitebox.yaml` source model stays beside the Markdown page, and generated complete and derived `.whitebox*.svg` files are stored under the page-local `assets/` directory; the complete SVG is linked from Markdown by default, while derived SVGs are linked only when they meet the reader-facing embedding policy. The first standard derived views are `boundary`, `delegation`, and `assembly`, with `interfaces` generated only when interface roles exist; empty derived views are not generated. Under the free and open-source constraint, ELK is the preferred Whitebox Layout Backend for automatic layout because it handles ports, compound nodes, labels, layered layout, and orthogonal routing while keeping layout results outside the source model. The enclosing component boundary is modeled as a real ELK compound node, not drawn afterward as an unaware SVG frame. The renderer does not pre-assign boundary port sides by default; ELK should retain freedom to place ports and route edges, with deterministic output coming from stable ids, stable ordering, and fixed backend options. If the preferred layout backend is missing or fails, rendering fails loudly instead of silently falling back to a lower-quality layout; the legacy simple backend may be used only when explicitly selected for diagnostics or migration comparison. Dependencies such as `elkjs` are installed only during development or upgrade of the Repo Wiki Skill Suite itself, not during ordinary Skill execution or diagram rendering. Layout quality is checked structurally, geometrically, and visually; pixel-perfect snapshot equality is not the only correctness signal. ELK owns geometry only; the suite's Semantic SVG Renderer remains responsible for the final notation, accessibility text, interface-role symbols, connector styling, and reader-facing SVG semantics.
